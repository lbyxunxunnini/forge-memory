# 反模式与禁忌

本文档汇总 forge-memory 使用中常见的错误用法和禁忌行为。用错场景比不用更糟——错误的索引会误导 AI 读错文件。

## 触发反模式

### 不应该完整触发的场景

| 场景 | 正确做法 |
|------|----------|
| 修改一个函数名 | 直接读文件修改，不触发 forge-memory |
| 看单个文件的 bug | 直接读文件，不触发扫描 |
| 格式化代码 | 不需要项目全貌，直接操作 |
| 运行单个测试 | 不需要索引，直接运行 |

**判断标准**：如果任务只需要读 1-3 个文件，不需要项目全貌，就不应该触发完整扫描。

### 完全不应该触发的场景

| 场景 | 原因 |
|------|------|
| 总结今天的会话 | 会话总结不需要项目扫描 |
| 无 `.project-context/` 且用户未要求初始化 | 没有索引可读，不应自动创建 |
| 用户在写文档而非写代码 | 文档任务通常不需要代码索引 |

## 使用禁忌

### 1. 被动读取时触发扫描/初始化

被动读取（三级触发）的目的是快速定位代码模块。**只读 `project-summary.md` 一个文件**，不得触发扫描、初始化或任何写入操作。

```
# 错误：被动读取时触发了扫描
用户："帮我看看 auth 模块的 bug"
→ 触发 forge-memory scan
→ 扫描整个项目

# 正确：被动读取时只读摘要
用户："帮我看看 auth 模块的 bug"
→ 读取 project-summary.md
→ 从摘要中定位 auth 模块路径
→ 定向读取源码
```

### 2. 覆盖用户手写摘要

`project-summary.md` 可能包含用户手写的业务上下文和决策记录。扫描时**不得擅自覆盖**，必须先询问。

```
# 错误：直接覆盖
scan() → 覆盖 project-summary.md → 用户手写内容丢失

# 正确：先询问
发现 project-summary.md 包含手写内容，是否覆盖？
```

### 3. 局部任务读取全部索引

修改单个模块时，不需要读取整个 `files.jsonl`（可能有数万行）。应该从 `project-summary.md` 的模块索引定位到目标模块，再定向读取。

### 4. 无错误提示地静默失败

当操作失败时（如备份恢复失败、进度保存失败），必须向用户报告。静默吞掉错误会导致用户以为操作成功，后续产生更难排查的问题。

### 5. 在非 git 项目中依赖分支功能

非 git 项目没有分支概念。`current_branch()` 会返回 `"default"`，commit 历史功能不可用。不要在非 git 项目中尝试使用 `impact` 或 git 相关分析。

## 配置反模式

### 排除目录不完整

只排除 `.git` 而不排除 `node_modules`、`__pycache__` 等目录，会导致扫描结果被大量无意义文件污染。

**推荐排除列表**（写入 `.forgeignore`）：
```
node_modules
__pycache__
.venv
venv
dist
build
.gradle
.cocoapods
```

### 扫描频率过高

每次代码修改后都运行 `scan` 是不必要的。forge-memory 支持增量扫描（基于 content_hash），但频繁扫描仍会浪费时间。

**推荐**：在以下时机扫描：
- 开始新功能开发前
- 合并分支后
- 项目结构发生重大变化后
- 上下文包质量下降时（`status` 会提示建议重新扫描）

## 诊断命令

遇到问题时，先运行诊断：

```bash
python3 forge_memory.py doctor <项目目录>
```

会检查：
- 环境依赖（git、python 版本）
- 索引完整性（文件覆盖率、新鲜度）
- 常见配置错误
