# Transition Gap Assessment

本评估回答一个问题：Forge Memory 做到什么程度后，用户从“智能体工作流设计者”转向“智能体平台工程师”的差距会明显缩小。

## 当前基线

| 能力 | 当前分 | 说明 |
|---|---:|---|
| 智能体工作流设计 | 85 | 已有任务驱动、领域 controller、门禁和验证闭环 |
| 技能/controller 产品化 | 82 | 多个技能项目已有发布和版本治理 |
| CLI 智能体原型 | 62 | 有真实代码和测试，但仍有类型和稳定性短板 |
| 项目记忆/上下文 | 52 | `project-memory` 有摘要能力，但不是任务级查询系统 |
| 数据和索引工程 | 35 | 缺少持久结构化索引和增量更新 |
| Git 历史和影响分析 | 30 | 当前主要是 worktree 快照 |
| MCP 产品契约 | 55 | 有 MCP 项目经验，但未成为核心上下文接口 |
| 平台可靠性 | 38 | 缺少完整降级路径、fixture 和系统测试 |

估算平台工程进度：

```text
58 / 100
```

## 完成 v0.1 后

预期变化：

```text
58 -> 64
```

原因：

- 证明能把项目扫描转成结构化索引。
- 证明能稳定生成任务级上下文包。
- 但还没有任务驱动框架真实集成和历史影响分析。

## 完成 v0.2 后

预期变化：

```text
64 -> 70
```

原因：

- 加入 Git 历史、文件变更和影响分析。
- 系统开始从”记忆文件”升级为”项目事实服务”。

## 完成 v0.3 后

预期变化：

```text
70 -> 73
```

原因：

- SQLite 可选索引证明静态存储的升级路径可行。
- 查询性能在大项目上有可感知提升。

## 完成 v0.4 后

预期变化：

```text
73 -> 76
```

原因：

- 任务驱动框架计划阶段开始消费上下文基础设施。
- 工作流设计能力和平台底座开始连接。

## 完成 v0.5 后

预期变化：

```text
76 -> 79
```

原因：

- 检索质量和轻量 RAG 提升上下文相关性。
- chunk metadata 和过期检测让上下文包更可靠。

## 完成 v0.6 后

预期变化：

```text
79 -> 80
```

原因：

- 可选本地 API 降低 CLI 启动成本。
- 多项目路由和后台同步成为可能。

## 完成 v1.0 后

预期变化：

```text
76 -> 82
```

定位变化：

```text
从：准备转型的智能体工作流设计者
到：有工作流优势的智能体平台工程师
```

## v1.0 后仍未闭合的差距

- 多用户/分布式平台运行。
- 生产级服务观测。
- 权限、密钥和隐私治理。
- 大仓库长期性能。
- RAG/embedding 质量评估。
- 部署、升级和回滚机制。

这些属于 v1.5/v2 能力，不阻塞第一阶段转型证明。
